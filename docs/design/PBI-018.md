# PBI-018: 動画解析プログレスバー表示の異常数値修正 - 詳細設計

## 1. 概要
動画解析中にプログレスバーの数値が 100% を大幅に超える（8000% など）事象を修正する。
この問題は、バックエンドがパーセンテージ (0-100) を送信しているのに対し、フロントエンドがそれを小数 (0-1) と解釈して 100 倍していることに起因する。

## 2. 影響範囲
### バックエンド (mhws-vision-server)
- `src/services/task_queue.py`: プログレス通知の値（パーセンテージ -> 小数）
- `src/db/database.py` (JobModel): DBに保存される値の形式（任意だが、APIと合わせるのが望ましい）

### フロントエンド (mhws-vision-client)
- `src/store/visionStore.ts`: 受信したプログレス値の処理
- `src/lib/sse-client.ts`: スタブシミュレーションの値（既に 0-1 なので維持）

## 3. 解決策 (SSoT: OpenAPI 準拠)
OpenAPI (`openapi.yaml`) の `AnalysisJobResponse.progress` は `number (float)` で定義されている。
一般的な API 設計慣習に従い、**プログレス値は 0.0 〜 1.0 の範囲の小数**として扱うよう統一する。

### バックエンドの変更
- プログレス計算時に `* 100.0` を行わない。
- `min(..., 0.99)` 等のクリッピングも 0.0 〜 1.0 の範囲で行う。

### フロントエンドの変更
- 現在の `data.progress * 100` のロジックは、以前の設計（小数を期待）に基づいているため、バックエンド修正後はそのまま機能する。
- 防御的コーディングとして、UI表示前に `Math.min(val, 100)` を適用する、または異常値のバリデーションを追加する。

## 4. ロジックフロー
1. バックエンド: `frames_processed / total_frames` を計算（0.0 〜 1.0）。
2. バックエンド: `/api/v1/analyze/events` (SSE) で `"progress": 0.1` 等を送信。
3. フロントエンド: `visionStore` が受信し、`0.1 * 100 = 10` を status にセット。
4. UI: `10%` と表示。
